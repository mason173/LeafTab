# LeafTab 搜索增强路线图

日期：2026-04-24

## 1. 这份文档要解决什么

这份文档不是代码审查，也不是性能整改清单。

它要回答的是更上层的问题：

- LeafTab 的搜索接下来应该往哪个方向增强
- 哪些能力是 `MVP`
- 哪些能力是 LeafTab 的 `关键差异`
- 哪些能力虽然很酷，但现在 `先别做`

目标不是把搜索做成“功能越来越多”。

目标是把 LeafTab 的搜索，从一个强搜索框，做成一个真正的：

`Browser-native Raycast`

也就是：

- 一个浏览器里的统一入口
- 一个浏览器里的命令面板
- 一个浏览器里的直接操作中心

## 2. 产品定位

### 2.1 我们不只是做“搜索”

LeafTab 的搜索不应该只负责：

- 帮用户搜到网页
- 帮用户搜到历史记录
- 帮用户搜到书签

它更应该负责：

- 帮用户切换到正确的浏览器对象
- 帮用户直接对这个对象执行操作
- 帮用户快速进入下一步工作状态

所以 LeafTab 搜索的核心价值，不是“查到什么”，而是：

`让用户更快完成浏览器里的下一步动作`

### 2.2 为什么这条路对 LeafTab 合适

因为 LeafTab 本来就不是一个普通的新标签页装饰插件。

它天然已经有这些基础：

- 新标签页入口
- 快捷方式系统
- 搜索面板
- 命令式交互能力
- 设置系统
- 视觉首页

这意味着它最适合做的，不是“再加几个搜索源”，而是把这些能力整合成一个统一入口。

## 3. 核心原则

后续所有搜索增强，都建议遵守下面 5 个原则。

### 3.1 统一入口

用户不应该先想：

- 我要去 tabs 面板
- 我要去 history 面板
- 我要去 bookmarks 面板

而应该先进入一个统一入口，再让系统帮用户找到最适合的结果。

### 3.2 结果可操作

搜索结果不应该只是“可点击项”。

搜索结果应该是：

- 标签页对象
- 书签对象
- 历史记录对象
- 快捷方式对象
- 设置项对象

既然是对象，就应该可以直接操作。

### 3.3 键盘优先

如果用户要不断从键盘切到鼠标，再切回来，搜索再强也很难有 Raycast 感。

所以搜索增强必须默认考虑：

- 上下选择
- 回车确认
- 快捷动作
- 不离开键盘的操作链

### 3.4 混排优先，分面板次之

LeafTab 未来更应该依赖：

- 智能混排

而不是继续堆很多独立面板。

分面板不是不能做，但它更适合作为辅助模式，而不是用户主路径。

### 3.5 不拿实时体验换“看上去更聪明”

搜索可以越来越强，但不能为了更复杂的逻辑牺牲：

- 输入响应
- 面板打开速度
- 结果切换速度
- 设置项的即时反馈

如果一个能力很强，但会明显让操作变钝，就不应该优先做。

## 4. 用户最核心的使用场景

LeafTab 搜索增强建议围绕下面几种高频场景设计。

### 4.1 找到正在进行中的东西

比如：

- 切到已经打开的标签页
- 打开刚刚访问过的页面
- 恢复最近关闭的内容

### 4.2 对浏览器对象直接执行动作

比如：

- 关闭某个标签页
- 固定某个标签页
- 删除某个书签
- 编辑某个快捷方式

### 4.3 进入某种工作状态

比如：

- 开始工作
- 恢复阅读
- 清理杂乱标签页
- 继续刚才的搜索任务

### 4.4 直接控制 LeafTab 自己

比如：

- 改搜索设置
- 改壁纸设置
- 改图标设置
- 切换主题模式

## 5. MVP

这一层的标准是：

做完以后，用户会明显感觉 LeafTab 搜索已经不是一个普通搜索框了。

### 5.1 统一混排搜索

同一个输入里直接混排：

- tabs
- bookmarks
- history
- shortcuts
- commands
- web search

不要要求用户先选数据源。

为什么这是 MVP：

- 它决定了搜索是不是“统一入口”
- 它是后续所有增强能力的地基

用户感知变化：

- 搜一个词，不用先想去哪个面板
- 更像命令面板，而不是多个面板的入口页

### 5.2 Tab 直接操作

搜索到标签页后，不只是切换过去，还应该允许直接操作。

第一版最值得做的动作：

- 切换
- 关闭
- 固定 / 取消固定
- 复制链接

为什么这是 MVP：

- tab 是浏览器里最高频的对象
- 用户最容易立刻感知价值
- 能最快把 LeafTab 搜索从“查找器”升级成“控制器”

用户感知变化：

- 搜索结果不再只是打开项
- 很多浏览器操作不用再先切到页面里再点

### 5.3 Bookmark 直接操作

搜索到书签后，建议第一版支持：

- 打开
- 删除
- 复制链接
- 在新标签页打开

为什么这是 MVP：

- 书签和 tab 一样，是非常典型的浏览器对象
- 直接操作会让搜索的“执行力”立刻变强

### 5.4 空输入推荐

当用户没有输入任何内容时，也应该给出高价值入口。

第一版推荐内容建议包括：

- 最近关闭标签页
- 最近访问页面
- 高频快捷方式
- 最近一次搜索续接

为什么这是 MVP：

- 新标签页天然就是“还没输入，但已经想做事”的场景
- 空状态如果有价值，搜索就会从被动工具变成主动入口

### 5.5 键盘优先的基础交互

最基本的一套键盘路径必须顺畅：

- 上下选择
- `Enter` 打开 / 执行动作
- `Esc` 关闭
- `Cmd/Ctrl + Enter` 在新标签打开
- 删除类快捷键执行删除 / 关闭动作

为什么这是 MVP：

- 没有键盘优先，就没有 Raycast 感
- 这是搜索体验是否高级的底层标准

## 6. 关键差异

这一层不是“能不能用”，而是“为什么用户会觉得 LeafTab 和别的产品不一样”。

### 6.1 搜索结果直接操作

这是 LeafTab 搜索最关键的差异点。

不是：

- 搜到结果
- 点进去
- 再找按钮

而是：

- 搜到结果
- 当场完成动作

典型例子：

- tab：切换、关闭、固定、复制链接
- bookmark：打开、删除、复制链接
- shortcut：打开、编辑、删除、移动
- setting：直接切换选项、直接改值

为什么它是关键差异：

- 它会让搜索从“结果列表”变成“操作中心”
- 用户最容易在这一步感受到 LeafTab 的高级感

### 6.2 浏览器对象优先

LeafTab 不应该去模仿一个泛化的桌面启动器。

它的优势在于“浏览器原生”。

所以最该做深的对象是：

- tabs
- bookmarks
- history
- shortcuts
- current page
- LeafTab settings

为什么它是关键差异：

- 这是 Raycast 在浏览器场景里做不到的深度
- 这是 LeafTab 最天然的护城河

### 6.3 个性化排序

同样搜 `notion`：

- 有些人想切到已打开 tab
- 有些人想打开收藏的书签
- 有些人想发起网页搜索

LeafTab 应该逐步学习用户偏好，让结果排序更像“懂我”。

建议优先学习的维度：

- 用户更常点哪类 source
- 哪些域名更高频
- 哪些动作更常被触发
- 某些时间段偏好的结果类型

为什么它是关键差异：

- 功能再多，都不如“结果一出来就是你想要的”
- 这会让产品有明显的长期使用黏性

### 6.4 工作流命令

LeafTab 不只要帮用户找东西，还应该帮用户进入某种状态。

比如：

- `/work`
- `/focus`
- `/read`
- `/resume`
- `/clean`

这些命令不只是打开一个页面，而是触发一组浏览器行为或一组推荐入口。

为什么它是关键差异：

- 这会让 LeafTab 真正变成新标签页工作台
- 用户会开始把它当“开始做事的入口”

### 6.5 设置项可搜索、可直接改

这个点很容易拉开与普通新标签页插件的差距。

例如用户搜：

- 壁纸
- 模糊
- 图标大小
- 搜索引擎
- 深色模式

理想状态不是只跳去设置页，而是：

- 直接切换
- 直接调整
- 直接预览

为什么它是关键差异：

- LeafTab 自己本身就是一个可配置工作台
- 搜索如果能直接控制设置，就会非常像“系统控制中心”

## 7. 很酷，但先别做

这一层的能力不是没有价值，而是现在做，性价比不高，甚至可能把产品做散、做慢。

### 7.1 大而全的自然语言 AI

比如一上来就想支持：

- “帮我整理今天的浏览记录”
- “总结我最近看过的网页”
- “自动理解所有自然语言命令”

这类方向很酷，但风险也很高：

- 语义边界不清晰
- 交互预期会迅速抬高
- 很容易把输入体验做慢

建议：

- 先做轻量规则型 intent
- 先把命令式搜索和直接操作做好

### 7.2 批量操作系统

比如：

- 多选多个结果
- 批量关闭标签页
- 批量移动书签
- 批量复制链接

这类功能很强，但不是第一阶段最重要的。

建议：

- 先把单结果动作打磨好
- 等单个对象的动作链顺畅后，再做批量系统

### 7.3 重型预览面板

比如在右侧加一个长期占位的大预览区，展示更多页面信息、截图、上下文等。

它不是没用，但现在优先级不高。

建议：

- 先做轻量上下文提示
- 先别引入重布局和重渲染结构

### 7.4 继续堆很多独立面板

例如：

- 命令面板
- tabs 面板
- history 面板
- bookmark 面板
- settings 面板

越拆越多，看起来功能更全，但用户心智会越来越碎。

建议：

- 主路径坚持统一入口 + 智能混排
- 分面板只作为辅助模式存在

### 7.5 重 AI 化的推荐与总结

比如自动分析用户浏览习惯、自动总结站点、自动推荐一整套复杂工作流。

这些方向后面可以探索，但现在不是最该优先建立护城河的部分。

## 8. 推荐的阶段顺序

如果按实际落地顺序，我建议这样排：

### Phase 1：把搜索做成真正的统一入口

- 统一混排搜索
- 键盘优先基础交互
- 空输入推荐

这阶段的目标是：

让 LeafTab 搜索先具备“统一入口”的产品感。

### Phase 2：把搜索结果做成可执行对象

- Tab 直接操作
- Bookmark 直接操作
- Shortcut 基础直接操作

这阶段的目标是：

让 LeafTab 搜索从“找东西”进化成“办事情”。

### Phase 3：做出 LeafTab 的明显差异

- 个性化排序
- 工作流命令
- 设置项直接操作

这阶段的目标是：

让用户真正觉得 LeafTab 搜索不是别人都能替代的。

### Phase 4：谨慎扩展更高级能力

- 轻量自然语言 intent
- 批量操作
- 轻量预览

这阶段的目标是：

在不伤害流畅度和主路径清晰度的前提下，增加高级能力。

## 9. 每一类能力对用户的感知变化

### 9.1 统一混排搜索

用户感知：

- “我不用先选面板了”
- “这个入口更像命令面板了”

### 9.2 直接操作

用户感知：

- “它不只是给我结果，它能直接帮我做事”
- “我不用先点进去再处理”

### 9.3 个性化排序

用户感知：

- “它越来越懂我”
- “我常用的结果越来越靠前”

### 9.4 工作流命令

用户感知：

- “它不是搜索框，是浏览器工作台入口”

### 9.5 设置项直接操作

用户感知：

- “LeafTab 可以像系统控制中心一样被搜索和控制”

## 10. 一句话版本的优先级结论

如果只用一句话来概括：

`先把 LeafTab 搜索做成真正的统一入口，再把结果做成可直接操作的对象，最后再做个性化和工作流。`

## 11. 最终建议

LeafTab 搜索的未来，不应该是：

- 一个功能越来越多的搜索框

而应该是：

- 一个浏览器里的统一入口
- 一个浏览器里的命令面板
- 一个浏览器里的对象操作中心

所以接下来最推荐优先做的 5 件事是：

1. 统一混排搜索
2. Tab 直接操作
3. Bookmark 直接操作
4. 空输入推荐
5. 个性化排序

如果这 5 件事做好，LeafTab 搜索就会从“已经很强”进入“真正有产品壁垒”的阶段。
