# LeafTab Grid Drag System Spec — Rebuild Baseline

## 0. 重构任务目标

我要你**重构**现有网格拖拽系统，而不是在旧实现上继续修补。

必须遵守：

* 不要在旧代码基础上打补丁式修改
* 先重建数据模型、状态机、纯函数引擎
* 再重建 UI 层
* 不要直接开始写 React 组件
* 如果发现规则冲突，先指出，不要擅自决定

最终目标：

* 构建一个**状态驱动 + 纯函数推导**的二维网格拖拽引擎
* React / DOM 只负责渲染结果，不负责决策
* 所有交互结果必须由 `DragSession` 推导得到

---

## 1. 架构总原则

### 1.1 唯一真相

* `sequence` 是唯一真相
* 所有排序、避让、插入、合并都基于 `sequence`
* 不允许基于几何方向直接决定最终排序结果

### 1.2 Z 字形序列

使用 **0-based 行索引**：

* `row % 2 === 0` → 左到右
* `row % 2 === 1` → 右到左

即：

* 第 0 行：左到右
* 第 1 行：右到左
* 第 2 行：左到右

### 1.3 交互层与视觉层分离

三层职责必须严格分离：

* `A 区域`：唯一交互层（拖拽、命中、判定）
* `icon`：视觉展示层（不参与任何逻辑计算）
* `shadow`：预测反馈层（仅表示 preview anchor）

任何实现不得混用三者职责。

### 1.4 纯函数原则

实现必须遵循：

1. 所有计算必须为纯函数
2. 不允许在事件回调中直接修改布局状态
3. `anchor / highlight / reorder / merge preview` 必须统一由 `DragSession` 推导
4. 不允许多个地方重复计算同一结果
5. React / DOM 只渲染，不做决策

### 1.5 排序语义

全系统统一使用：

* `remove + insert + shift`

禁止：

* swap
* nearest-cell swap
* DOM 顺序驱动排序
* 基于视觉位置直接改真实顺序

---

## 2. 网格系统定义

### 2.1 网格基础

这是一个二维平面网格系统：

* 网格整体在一个大容器中
* 网格数量不受容器宽度影响
* 用户可调节为 `N × M`
* 每个网格始终为正方形
* 网格之间没有间隙（gap = 0）
* 容器缩放时，网格整体等比缩放
* 排列方式为 Z 字形流动，不是普通 row-major

### 2.2 网格单元结构

每个网格包含 4 层：

1. 网格本身
2. 容器 A（位于中心偏上，大小为 cell 的 80%）
3. icon（大小独立，不影响布局）
4. icon 下方有 shadow + 文本

### 2.3 拖拽基础规则

* 只有点击容器 A 才能触发拖拽
* 拖拽判定点 = 容器 A 的中心点
* 拖动时跟随移动：A + icon + 文本
* shadow 不跟随 icon 实时移动
* shadow 表示 preview anchor

---

## 3. 视觉尺寸与布局规则

### 3.1 icon 尺寸规则

每个网格内的 icon 大小由用户单独控制，与网格尺寸完全解耦。

规则：

* icon 的视觉尺寸不随 `cellSize` 自动缩放
* 容器缩放只影响网格，不影响 icon 大小
* icon 可以大于或小于容器 A
* icon 不参与布局计算
* icon 尺寸变化不会影响：

  * sequence
  * 命中判定
  * 拖拽逻辑
  * 避让逻辑

禁止：

* 不允许使用 icon bounding box 参与命中计算
* 不允许根据 cellSize 自动缩放 icon
* 不允许 icon 尺寸影响排序逻辑
* icon 尺寸变化只触发视觉重绘，不触发布局重排

### 3.2 icon 在 A 内的位置

* icon 在容器 A 内水平居中
* icon 在容器 A 内垂直居中
* `iconCenter = center(A)`

允许存在一个固定视觉偏移，但该偏移：

* 不影响拖拽判定
* 不影响命中计算
* 不影响 anchor 计算

---

## 4. shadow（预期落点反馈层）

### 4.1 绑定关系

* shadow 的尺寸、形状、缩放与 icon 完全绑定
* shadow 不与 A 绑定
* icon 尺寸变化时，shadow 同步变化

### 4.2 拖拽时行为

拖拽时：

* icon 本体跟手移动
* shadow 不跟随 icon 实时移动
* shadow 表示“当前预期落点”

即：

* icon = 当前拖拽位置
* shadow = preview anchor 对应位置

### 4.3 位置计算

`shadowPosition = position(anchorSeq)`

说明：

* `position(anchorSeq)` 表示根据当前 Z 序列映射到网格坐标
* shadow 始终渲染在该位置
* shadow 的更新完全依赖 `previewAnchor`
* 不允许基于 icon 当前视觉位置计算 shadow

### 4.4 shadow 何时显示

#### 普通插入 / 普通换位

* 显示 shadow

#### 合并预期 / 进入文件夹预期

* 不再显示普通插入 shadow
* 由目标高亮承担主要反馈

### 4.5 唯一性

* 任意时刻，一个拖拽节点只能存在一个有效 preview anchor
* 任意时刻，一个拖拽节点只能存在一个有效 shadow 预期

## 5. 节点类型

系统有三种外部节点：

1. `icon`
2. `smallFolder`
3. `bigFolder`

以及一种内部使用场景：

4. 文件夹展开后的内部图标

## 6. 拖拽模式

系统有三种拖拽模式：

### 6.1 Normal

适用于普通图标：

* 可排序
* 可避让
* 可成组
* 可合并进入大文件夹

### 6.2 Reorder-only

适用于：

* smallFolder
* bigFolder
* 文件夹内部图标

特点：

* 只允许排序 / 避让
* 不允许成组
* 不允许合并

### 6.3 External-Insert

适用于从文件夹拖出到桌面时：

* 一进入外部网格就建立 preview anchor
* 允许排序 / 避让
* 不允许成组
* 不允许合并
* 不允许进入任何文件夹

---

## 7. 普通图标规则

普通图标占 1×1：

支持：

* 排序
* 避让
* 成组
* 合并进 bigFolder

### 7.1 普通图标成组规则

普通图标与普通图标成组后生成 `smallFolder`。

规则：

* 以目标位置为准
* A 拖到 C → C 位置变为文件夹
* A 从原位置移除
* 文件夹内容为 `[A, C]`

---

## 8. 边进入判定规则（核心）

此规则**只适用于 Normal 模式下的普通图标**。

判定必须同时看两件事：

1. 目标节点的固定方位归属（上 / 下 / 左 / 右）
2. 拖拽中心点从目标节点当前 A 区域的哪条边进入

### 8.1 方位归属冻结规则

1. 当 `dragStart` 发生时，必须以被拖拽节点的原始位置为参考，
   一次性确定所有其他节点相对于它的方位归属：

   * `upperTargets`
   * `lowerTargets`
   * `leftTargets`
   * `rightTargets`

2. 这些方位归属在整个 `DragSession` 生命周期内必须保持固定且只读。

3. 不允许因为以下因素重新计算或改变方位归属：

   * 预览避让
   * 链式位移
   * 动画过程
   * 视觉位置变化
   * 当前 DOM 几何
   * preview layout
   * 最近距离 / 当前最近邻

4. `activeTarget` 可以动态变化，但它属于上 / 下 / 左 / 右哪个集合，
   必须始终读取 `dragStart` 时冻结的数据。

### 8.2 方位与进入边规则

#### 上方目标

* 从右 / 下进入 → `group-candidate`
* 从左 / 上进入 → `reorder-candidate`

#### 下方目标

* 从左 / 上进入 → `group-candidate`
* 从右 / 下进入 → `reorder-candidate`

#### 同一行左侧目标

* 从右 / 下进入 → `group-candidate`
* 从左 / 上进入 → `reorder-candidate`

#### 同一行右侧目标

* 从左 / 上进入 → `group-candidate`
* 从右 / 下进入 → `reorder-candidate`

### 8.3 强约束与说明

* 最终行为判定必须使用：
  `固定方位归属 + 当前进入边`
* 方位归属是静态冻结数据，只能来自 `dragStart`
* `enterEdge` 是动态数据，必须基于目标节点当前时刻的 A 区域边界实时计算
* `group-candidate` 不等于立即合并，只是进入可合并状态
* `reorder-candidate` 表示普通排序 / 避让预期
* `group-candidate` 在拖拽中心点进入目标 A 区域并满足边进入规则时立即触发
* 不允许把这套规则简化成 overlap / nearest / distance-only 判定
* 不允许在拖拽过程中基于当前布局、当前 DOM 位置或 preview 结果重新推导目标属于上 / 下 / 左 / 右

## 9. activeTarget 规则（必须实现）

系统必须维护一个 `activeTarget`。

### 9.1 锁定规则

1. 当拖拽中心点首次进入某个普通目标节点的 A 区域时：

   * `activeTarget = 该节点`

2. 在 `activeTarget` 未被释放前：

   * 所有边进入判定
   * 所有成组判断
   * 所有普通合并判断
     都必须基于 `activeTarget`
   * 不得每一帧重新通过 globalPosition 改选普通目标
   * `activeTarget` 的方位归属必须读取冻结后的 `directionMap`，
     不允许根据它当前视觉位置重新判断是上 / 下 / 左 / 右

3. 只有在以下情况才允许释放 `activeTarget`：

   * 拖拽中心点离开当前 `activeTarget` 的 A 区域
   * 或进入更高优先级命中区域（例如 `bigFolderMergeHitArea`）

4. `activeTarget` 一旦释放，才允许重新命中新目标

### 9.2 约束

* `activeTarget` 必须是 `DragSession` 的推导结果
* 它不是事件里的临时副作用变量
* 任意时刻最多只能有一个 `activeTarget`
* `activeTarget` 可变化，但节点的方位身份不跟随 `activeTarget` 的当前几何位置变化
* 重新命中旧目标时，仍然必须沿用该目标在 `dragStart` 时冻结的方位归属

---

## 10. smallFolder 规则

smallFolder 占 1×1。

支持：

* 排序
* 避让
* 视觉上看起来可换位

不支持：

* 成组
* 合并
* 进入任何文件夹

### 10.1 重要语义

smallFolder 的“换位”只是视觉结果，不代表算法是 swap。

实现必须使用：

* `remove + insert + shift`

禁止：

* swap(a, b)

---

## 11. bigFolder 规则

bigFolder 占 2×2。

### 11.1 静止状态

* bigFolder 是固定障碍物
* 不可被挤走
* 其他节点不能占用其区域
* 链式避让遇到静止 bigFolder 时，必须跳过其完整 footprint

### 11.2 拖动状态

bigFolder 拖动时必须作为一个整体 block 处理，不能拆成 4 个独立节点。

使用：

* `center-based 2×2 block insert`

### 11.3 bigFolder 锚点策略

固定使用：

* `bigFolderAnchorPolicy = "center"`

说明：

* bigFolder 的 center anchor 是逻辑锚点
* 不要求对应单一真实 cell 的几何中心
* 它的职责是稳定地映射到唯一的 2×2 footprint
* 同一 `globalPosition + gridSpec` 必须推导出唯一且稳定的 footprint

### 11.4 footprint 合法性

bigFolder 的 footprint 必须满足：

* 不得越出 grid 边界
* 不得形成不完整 2×2 区域

若推导出的 footprint 非法：

* 要么自动调整到最近合法位置
* 要么标记为 invalid，不允许插入

### 11.5 bigFolder 链式避让规则

bigFolder 插入时：

1. 先根据 center anchor 推导 2×2 footprint
2. 收集被 footprint 覆盖的冲突节点
3. 冲突节点必须按当前 sequence 升序收集
4. 再整体沿 `sequence.next` 后移
5. 最后 bigFolder 占据该 footprint

禁止：

* 按屏幕几何顺序处理
* 按 DOM 顺序处理
* 按 2×2 遍历顺序处理
* 覆盖或拆分静止 bigFolder

---

## 12. bigFolder 合并规则

普通图标进入 bigFolder 时：

* 命中区域使用 `bigFolderMergeHitArea`
* 不复用普通 1×1 节点的 A 区域逻辑
* 不区分进入边
* 中心点进入命中区域 → `merge-candidate`

### 12.1 用户可见效果

进入 bigFolder mergeHitArea 时：

* bigFolder 整体高亮
* 不再显示普通插入 shadow
* 显示“将被收纳”预期

松手：

* 执行合并

离开：

* 取消高亮
* 恢复普通排序预期

---

## 13. 文件夹展开规则

文件夹展开后，内部应视为一个独立网格系统。

继承：

* Z 字形排序
* 拖拽规则
* A 区域命中
* 避让逻辑

限制：

* 不允许成组
* 不允许合并
* 不允许生成新文件夹
* 不允许嵌套文件夹

本质：

* 文件夹内部拖拽 = `Reorder-only`

---

## 14. 文件夹自动解散

* 文件夹只剩 1 个图标 → 自动解散
* 不存在 0 个图标的文件夹

---

## 15. 从文件夹拖到桌面

当图标从文件夹内部拖向桌面：

### 15.1 进入外部网格时

* 立即建立 `previewAnchor`
* 进入 `External-Insert` 模式

### 15.2 External-Insert 规则

允许：

* 排序
* 避让

禁止：

* 成组
* 合并
* 进入任何文件夹

### 15.3 松手后

* `anchor → index`
* 节点正式成为外部普通图标
* 后续新的拖拽中才重新恢复 Normal 能力

## 16. 合并预期反馈（白色 border）

当拖拽节点进入“可合并状态”时，目标节点必须显示视觉高亮。

### 16.1 高亮对象

高亮始终加在目标节点上，而不是拖拽节点上：

* icon → 高亮该 icon
* smallFolder → 不应进入合并高亮（因不接受合并）
* bigFolder → 高亮整个 2×2 区域

### 16.2 高亮样式

* 使用白色 border（或等效高亮）
* 包裹目标节点的视觉边界
* 不影响布局
* 不参与命中计算

### 16.3 触发条件

当满足以下任一条件时触发：

1. Normal 模式下：

   * 命中目标节点 A 区域
   * 满足边进入规则
   * 进入 `group-candidate`

2. 普通图标进入 `bigFolderMergeHitArea`

### 16.4 取消条件

以下任一情况必须立即取消高亮：

* 离开目标命中区域
* 当前判定从 `group-candidate → reorder-candidate`
* 进入 Reorder-only
* 进入 External-Insert
* 松手结束拖拽

### 16.5 唯一性

* 任意时刻最多只能存在一个高亮目标
* 不允许多个目标同时高亮

### 16.6 与 shadow 的关系

* `shadow = 插入预期`
* `border = 合并预期`

同一时刻，两者不能表达同一语义。

当出现合并高亮时：

* 普通插入 shadow 不再作为主要反馈

---

## 17. 无命中与离开命中规则

### 17.1 无命中状态

当拖拽中心点：

* 未命中任何普通节点 A 区域
* 且未命中任何 bigFolder 的 mergeHitArea

则：

* 不触发成组
* 不触发合并
* 不显示目标高亮
* 拖拽节点仅保持跟手移动
* 若存在合法普通 preview anchor，则显示普通 shadow
* 若不存在合法 preview anchor，则 shadow 可隐藏或标记 invalid

### 17.2 离开目标状态

当拖拽中心点离开当前目标命中区域时：

* 当前目标高亮立即取消
* 当前合并预期立即取消
* 恢复普通插入 / 普通排序预览
* 所有 preview 结果必须基于新的 anchor 重新计算

### 17.3 重要说明

拖拽过程中没有任何节点“真正占位”，只有 preview 占位。

也就是说：

* 拖拽中看到的避让 / 腾空 / shadow 都是预览结果
* 只有松手时才提交为真实 sequence

---

## 18. 链式避让规则

### 18.1 总原则

所有避让都必须沿 `sequence.next` 方向执行。

不得：

* 基于左 / 右 / 上 / 下几何方向决定最终避让方向
* 基于视觉最近位置决定最终顺移

### 18.2 普通 1×1 节点

对于：

* icon
* smallFolder
* 文件夹内部图标
* External-Insert 阶段的拖出图标

排序语义固定为：

* `remove + insert + shift`

### 18.3 bigFolder

对于 bigFolder：

* 先根据 center anchor 推导 2×2 footprint
* 收集 footprint 覆盖的冲突节点（按 sequence 升序）
* 再整体沿 `sequence.next` 后移
* 静止 bigFolder 为不可穿透障碍物，必须跳过完整 footprint

### 18.4 非法情况

如果最终无法找到合法空间：

* 当前插入无效
* `previewAnchor.valid = false`
* 不提交

---

## 19. 自动滚动与全局坐标

系统必须支持拖拽到边界自动滚动。

### 19.1 全局坐标

拖拽位置必须基于：

* `globalPosition = mousePosition + scrollOffset`

### 19.2 自动滚动规则

自动滚动时：

* `mousePosition` 不变
* `scrollOffset` 变化
* `globalPosition` 连续变化
* 所有 preview anchor 必须基于新的 `globalPosition` 重算
* preview 不能跳变

### 19.3 重要约束

自动滚动只改变：

* viewport
* scrollOffset

自动滚动不改变：

* sequence
* 网格世界坐标系
* 节点逻辑顺序

### 19.4 禁止行为

* 禁止直接使用 DOM rect 计算 targetIndex
* 禁止只根据当前可视区域决定插入位置
* 禁止滚动后只改视觉不重算 anchor

---

## 20. DragSession：唯一输入模型

所有核心计算必须从同一个输入模型推导。

```ts
DragSession = {
  globalPosition,
  mode,
  activeNode,
  activeTarget,
  directionMap: {
    upper,
    lower,
    left,
    right
  },
  gridSpec,
  occupancy
}
```

所有输出都必须是计算结果，而不是事件回调副作用：

* `anchorPreview = f(DragSession)`
* `highlightTarget = f(DragSession)`
* `reorderPreview = f(DragSession)`
* `shadowPosition = f(anchorPreview)`

重点：

* 不是“发生了什么就做什么”
* 而是“当前状态是什么，就推导出结果”
* `directionMap` 必须只在 `dragStart` 初始化一次
* `directionMap` 在整个拖拽会话中必须保持 immutable
* 后续预览位移、避让、动画都不能修改 `directionMap`

## 22. 必须实现的纯函数（最小集合）

至少需要定义并实现这些纯函数：
