## 一、系统概述

这是一个二维平面网格系统，特点：

* 网格整体在一个大容器中
* 网格数量不受容器宽度影响
* 用户可调节为 N × M
* 每个网格始终为正方形
* 网格之间没有间隙
* 容器缩放时，网格整体等比缩放
* 排列方式为 Z 字形流动（不是普通行列）

Z 字形规则：

* 偶数行：从左到右
* 奇数行：从右到左
* 所有排序必须基于 sequence index（next / prev），而不是简单几何方向

---

## 二、网格单元结构

每个网格包含 4 层：

1. 网格本身
2. 容器 A（位于中心偏上，大小为 cell 的 80%）
3. icon（大小独立，不影响布局）
4. icon 下方有 shadow + 文本

拖拽规则：

* 只有点击容器 A 才能触发拖拽
* 拖拽判定点 = 容器 A 的中心点
* 拖动时跟随移动：A + icon + shadow + 文本

---

## 三、节点类型

系统有三种节点：

1. 普通图标（icon）
2. 小文件夹（smallFolder）
3. 大文件夹（bigFolder）

---

## 四、普通图标规则

普通图标占 1×1：

支持：

* 排序
* 避让
* 成组
* 合并进文件夹

---

## 四点五、边进入判定规则（核心）

普通图标在 Normal 模式下，必须同时判断：

1. 目标节点相对方位（上 / 下 / 左 / 右）
2. 拖拽中心点从目标节点的哪条边进入

规则：

上方目标：

* 右 / 下 = 进组
* 左 / 上 = 避让

下方目标：

* 左 / 上 = 进组
* 右 / 下 = 避让

同一行左侧目标：

* 右 / 下 = 进组
* 左 / 上 = 避让

同一行右侧目标：

* 左 / 上 = 进组
* 右 / 下 = 避让

说明：

* 进组 ≠ 立即合并，只是进入可合并状态
* 避让 = 按 Z 序列触发位置移动
* 该规则仅适用于 Normal 模式

---

## 五、普通图标成组规则

普通图标与普通图标成组后生成小文件夹：

规则：

* 以目标位置为准
* A 拖到 C → C 位置变为文件夹
* A 从原位置移除
* 文件夹内容为 [A, C]

---

## 六、小文件夹规则

小文件夹占 1×1：

支持：

* 排序
* 避让
* 换位

不支持：

* 成组
* 合并
* 进入任何文件夹

---

## 七、大文件夹规则

大文件夹占 2×2：

静止：

* 固定障碍物
* 不可被挤走
* 其他节点不能占用其区域

拖动：

* 作为整体 block（不能拆）
* 使用“整体插入 + 挤出”

规则：

* 目标区域节点按 Z 序列整体后移（next）
* 大文件夹插入

---

## 八、大文件夹合并规则

普通图标进入大文件夹：

* 中心点进入命中区域 → 可合并
* 不区分进入边

状态：

* 显示文件夹吸附预览
* 不显示普通排序占位

松手：

* 执行合并

离开：

* 恢复排序状态

---

## 九、文件夹展开规则

文件夹内部：

继承：

* Z 字形排序
* 拖拽规则
* 避让逻辑

限制：

* 不允许成组
* 不允许合并
* 不允许生成新文件夹

👉 本质：Reorder-only 模式

--------------------------------
十五、自动滚动与全局坐标规则
--------------------------------

系统必须支持拖拽到边界自动滚动。

关键要求：

1. 拖拽位置必须使用“全局坐标”计算：
   globalPosition = mousePosition + scrollOffset

2. 所有落点（preview anchor）必须基于全局网格计算，而不是当前可视区域

3. 自动滚动时：
   - mousePosition 不变
   - scrollOffset 变化
   - globalPosition 连续变化

4. preview anchor 必须连续更新，不能跳变

5. 禁止使用 DOM 位置或可视区域计算 index

否则会导致：
- 落点错位
- 拖拽不稳定
- 用户体验崩坏
再补 5 个实现级约束：

1. “所有命中基于 A 区域”这句话仅适用于普通 1x1 节点。
   对 bigFolder，普通图标进入其合并逻辑时，命中区域应单独定义为 bigFolderMergeHitArea，
   不要复用普通节点 A 区域语义。

2. bigFolder 的 center anchor 是逻辑锚点，不要求对应单一真实 cell 中心。
   它的职责只是稳定地映射到唯一的 2×2 footprint。
   同一 globalPosition + 同一 gridSpec，必须推导出唯一且稳定的 footprint。

3. applyBigFolderBlockInsert 时，
   被 2×2 footprint 覆盖的冲突节点必须按当前 sequence 升序收集，
   再按 stable insert 语义整体后移。
   禁止按屏幕几何顺序、DOM 顺序或 footprint 遍历顺序处理。

4. Normal 模式边进入判定里的“reorder”统一改名为 “reorder-candidate”，
   与 “group-candidate” 保持同一候选态语义。

5. 自动滚动只改变 viewport / scrollOffset，
   不改变 sequence，不改变网格世界坐标系。

## 图标尺寸规则（独立控制）

每个网格内的 icon 大小由用户单独控制，且与网格尺寸完全解耦。

规则如下：

- icon 的视觉尺寸不随 grid 的 cellSize 自动缩放
- grid 的缩放（例如容器宽度变化）只影响网格本身，不影响 icon 大小
- icon 可以大于或小于容器 A，不参与布局计算
- icon 尺寸变化不会影响：
  - sequence
  - 命中判定
  - 拖拽逻辑
  - 避让逻辑
- 拖拽判定始终基于容器 A 的中心点，而不是 icon 的视觉边界

说明：

- 容器 A 是交互层（命中与拖拽）
- icon 是纯视觉层（展示）

禁止：

- ❌ 不允许使用 icon 的 bounding box 参与命中计算
- ❌ 不允许根据 cellSize 自动缩放 icon
- ❌ 不允许 icon 尺寸影响网格布局或排序逻辑
icon 的尺寸变化只触发视觉重绘，不触发布局重排。
---

## 影子 S（落点反馈层）

每个 icon 下方存在一个与 icon 尺寸和形状完全一致的影子 S。

影子 S 的规则如下：

### 1. 绑定关系

- 影子 S 的尺寸、形状、缩放完全跟随 icon
- 不与容器 A 绑定
- icon 尺寸变化时，影子同步变化

---

### 2. 拖拽时行为

当 icon 被拖起后：

- icon 本体跟随拖拽移动
- 影子 S 不跟随 icon 实时移动
- 影子 S 表示“当前预期落点位置（preview anchor）”

即：

- icon = 当前拖拽位置（手指/鼠标）
- shadow S = 最终落点位置（系统计算）
